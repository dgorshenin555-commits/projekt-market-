# Личный кабинет — дизайн (спецификация)

**Дата:** 2026-06-18
**Платформа:** «Функция» (project-market), Next.js (App Router, static export), TypeScript, vanilla CSS (`.fx`), localStorage-моки.
**Статус:** утверждён к реализации (брейншторм согласован с пользователем).

## 1. Цель

Превратить тонкий `/dashboard` (сейчас обслуживает только заказчика и проектировщика) в **полноценный личный кабинет с вкладками**, разный по ролям, и убрать дублирование с «Настройками». Кабинет — единое место «про меня»: обзор, мои заявки/отклики/проекты, избранные нормативы, профиль.

## 2. Объём

**В фокусе** — два архетипа кабинета:
- **Заказчик** (`customer`).
- **Исполнитель** (`designer` = проектировщик / `expert` = инженер-обследователь) — один шаблон, данные зависят от подроли.

**Вне объёма (YAGNI):**
- Производитель (`manufacturer`) — отдельный кабинет позже; сейчас показываем минимальный fallback (обзор + профиль), не проектируем под него вкладки.
- Реальные бэкенд-функции уведомлений, документооборота, финансов — заглушки.
- Чат/аналитика/каталоги остаются отдельными разделами, в кабинет не втягиваются.

**Сквозное требование:** изоляция данных по пользователю + задел на права доступа — см. §16.

## 3. Информационная архитектура

- **Один пункт сайдбара «Личный кабинет»** заменяет текущий пункт «Настройки». Маршрут — существующий **`/dashboard`** (чтобы не плодить роуты и не ломать пост-логин-редирект `auth → /dashboard`).
- Иконка пункта — `user` (из набора `_orders/icons`).
- Внутри кабинета — **горизонтальные вкладки** (компонент в стиле существующих `.fx .tabs`). Активная вкладка хранится в локальном состоянии; опционально синхронизируется с query-параметром `?tab=` для прямых ссылок (например, `?tab=profile`).
- **`/settings`**: пункт сайдбара убираем. Страница-роут остаётся как тонкий редирект на `/dashboard?tab=profile` (чтобы не ломать существующие ссылки на `/settings`). Форма профиля выносится в общий компонент и переиспользуется вкладкой «Профиль».
- **Точки входа в кабинет:** пункт сайдбара «Личный кабинет» + клик по аватару/имени пользователя в топбаре (и в подвале сайдбара, если есть) ведут на `/dashboard`.

### Шапка кабинета (общая для всех ролей)
Аватар (инициалы) · имя/компания · бейдж роли · один ключевой показатель справа:
- заказчик → «N активных заявок»;
- исполнитель → рейтинг (★) и «N проектов в работе».

## 4. Вкладки — ЗАКАЗЧИК (`customer`)

1. **Обзор**
   - KPI-плитки (`.fx .stat`/`grid`): Активные заявки · В работе · Завершено · Всего откликов.
   - Блок «Последние отклики на мои заявки» (свежие `OrderResponse` по моим заявкам, до 3–5).
   - CTA «Создать заявку» (`/orders/new`).
   - Пустое состояние: если заявок нет — заглушка с CTA «Создать первую заявку».
2. **Мои заявки**
   - Сегмент-переключатель **«В работе» / «Отработанные»** (правила в §6).
   - Карточки заявок (переиспользуем стиль карточек dashboard/orders) → клик ведёт в `/orders/detail?id=…`.
   - Бейдж статуса, число откликов, бюджет, срок.
3. **Отклики**
   - Все отклики по всем моим заявкам, сгруппированные по заявке, чтобы сравнивать исполнителей и выбирать (`selectExecutor`).
   - У каждого отклика: исполнитель, сообщение, предложенные бюджет/срок, действия «Профиль» / «Выбрать».
4. **Уведомления** *(заглушка «скоро»)* — лента событий (новый отклик, смена статуса). Статичный аккуратный плейсхолдер.
5. **Профиль** — общий `ProfileForm` (см. §9) + блок «Аккаунт» (email, выход).

## 5. Вкладки — ИСПОЛНИТЕЛЬ (`designer` / `expert`)

1. **Обзор**
   - KPI: Мои отклики · В работе · Завершено · Рейтинг.
   - Блок «Рекомендованные заявки/обследования по моим разделам» (фильтр публичных заявок/обследований по специализации пользователя).
   - CTA: проектировщик → «Смотреть заявки» (`/orders`); обследователь → «Смотреть обследования» (`/expertise`).
2. **Мои отклики**
   - Проектировщик: мои `OrderResponse` со статусом заявки.
   - Обследователь: мои отклики на обследования (см. §11 — модель данных).
3. **В работе**
   - Проекты, где меня выбрали исполнителем (`order.assignedDesignerId === user.id`), с сегментом **«В работе» / «Завершённые»** (§6) и краткой полосой прогресса по статусу.
4. **Избранные документы**
   - Нормативы, помеченные ⭐ в разделе «Нормативы» (§7). Карточки/строки документа с быстрым переходом в «Нормативы».
   - Пустое состояние: «Пока нет избранных. Отметьте документы ⭐ в разделе Нормативы».
5. **Уведомления** *(заглушка «скоро»)*.
6. **Профиль и портфолио**
   - Публичная карточка: специализация (разделы ПД для проектировщика / виды обследования для обследователя), СРО, рейтинг, портфолио проектов (мок).
   - Общий `ProfileForm` (§9) + «Аккаунт».

**Различие подролей исполнителя** инкапсулируется в источнике данных и подписях: «заявки/`/orders`/разделы ПД» (designer) против «обследования/`/expertise`/виды обследования» (expert). Каркас вкладок один.

## 6. Разделение по статусу (сегмент «В работе» / «Отработанные/Завершённые»)

Применяется к «Мои заявки» (заказчик) и «В работе» (исполнитель). Маппинг `OrderStatus`:
- **«В работе» (активные):** `published`, `in_progress`.
- **«Отработанные/Завершённые»:** `completed`. `cancelled` — в «Завершённых» с пометкой «Отменена» (или скрыты — уточняется в плане; по умолчанию показываем в завершённых).
- `draft` — у заказчика в активных (как незавершённый черновик), если такие появятся; сейчас в моках нет.
Реализация — сегмент-контрол над списком (без отдельных под-роутов), фильтрация массива по бакету.

## 7. Избранные документы — модель и поток данных

**Проблема:** сейчас ⭐ в `/standards` — локальный `useState`-`Set`, инициализируемый из `isFeatured`; не сохраняется и не виден кабинету.

**Решение:** поднять избранное в общий стор с персистом в localStorage.
- В `AppState`/persisted-стейт добавить `favoriteStandards: string[]` (коды документов).
- В сторе — действие `toggleFavoriteStandard(code)` и геттер `getFavoriteStandards()` (возвращает `StandardDocument[]` по кодам из `MOCK_STANDARDS`).
- Инициализация: при первом запуске сидируем из `MOCK_STANDARDS.filter(isFeatured)` (сохраняем текущее поведение).
- `/standards` переводим с локального `favorites` на стор (`favoriteStandards` + `toggleFavoriteStandard`); вся остальная логика страницы (фильтр «Избранное», drawer) сохраняется.
- Кабинет (вкладка «Избранные документы») читает `getFavoriteStandards()`.
- Декоративная полоса `FavStrip`/`MOCK_FAVORITES` в `/standards` — синхронизируется со стором (показывает реальное избранное) либо остаётся декоративной; решаем в плане, по умолчанию — оставляем как есть, не трогаем.

## 8. Уведомления (заглушка)

Вкладка-плейсхолдер у обеих ролей: статичная лента из 3–5 примеров событий (новый отклик, заявка перешла «В работу», выбран исполнитель), пометка «Раздел в разработке». Без действий и без данных из стора.

## 9. Профиль — перенос из «Настроек»

- Текущую форму из `app/(internal)/settings/page.tsx` (роль, ФИО/компания, телефон, регион, условный СРО, «о себе», кнопка «Сохранить», индикатор «Сохранено», блок «Аккаунт» c email+выход) выносим в **переиспользуемый компонент** (напр. `app/_orders/ProfileForm.tsx` или `_cabinet/`).
- Вкладка «Профиль» в кабинете рендерит этот компонент. `/settings` → редирект на `/dashboard?tab=profile`.
- Логика `updateUser`/`logout`/синхронизация из localStorage сохраняется дословно.

## 10. Изменения модели данных и стора

- `AppState` (persisted): `+ favoriteStandards: string[]`.
- Стор: `+ toggleFavoriteStandard(code)`, `+ getFavoriteStandards()`.
- **Сторона обследования (решено — делаем реальной):** добавить мок `MOCK_EXPERTISE_RESPONSES` (отклик инженера-обследователя на обследование: `id`, `expertiseId`, `expertId`, `message`, `proposedBudget?`, `proposedDeadline?`, `createdAt`) + лёгкий тип `ExpertiseResponse` в `lib/types.ts`. Геттеры по аналогии с заявками: `getMyExpertiseResponses()` (мои отклики на обследования) и признак «назначен исполнителем обследования» для вкладки «В работе» (поле `assignedExpertId` на `ExpertiseRequest` либо аналог).

## 11. Что реально / что заглушка (честно)

**Реально (на данных/действиях):**
- Заказчик: Обзор, Мои заявки (со статусным сплитом), Отклики, Профиль.
- Проектировщик: Обзор, Мои отклики (`getMyResponses`), В работе (`assignedDesignerId`), Избранные документы (после §7), Профиль.
- **Обследователь (`expert`): Обзор, Мои отклики, В работе** — делаем реальными на мини-данных (`MOCK_EXPERTISE_RESPONSES`, см. §10), по аналогии с проектировщиком; Избранные документы и Профиль — как у всех.

**Заглушка / частично мок:**
- Портфолио/рейтинг исполнителя — мок.
- «Уведомления» — заглушка обеих ролей.

## 12. Компоненты и переиспользование

- Страница `/dashboard` → контейнер кабинета: шапка + `TabBar` + переключение контента вкладок по роли.
- Вкладки — отдельные компоненты (Overview/Orders/Responses/InWork/Favorites/Notifications/Profile), чтобы файл не разрастался; каждая — одна ответственность.
- Переиспользуем существующее: дизайн-систему `.fx` из `app/_orders/orders.css`, `Icon` (`_orders/icons`), `Avatar`, `StatusBadge`, карточки заявок, стор-геттеры (`getMyOrders`, `getMyResponses`, `orders`, `selectExecutor`, `updateUser`, `logout`).
- Никакого нового глобального CSS-фундамента сверх необходимого; добавки — точечно и под `.fx`.

## 13. Edge cases

- **Не авторизован:** контейнер кабинета показывает существующую заглушку «Необходима авторизация» + кнопку «Войти» (как сейчас в dashboard/settings). Все вкладки за этим гейтом.
- **Производитель/неизвестная роль:** fallback — Обзор (минимальный) + Профиль; остальные вкладки скрыты.
- **Пустые состояния** для каждой списочной вкладки (нет заявок/откликов/проектов/избранного) — с осмысленным CTA.
- **Прямая ссылка `?tab=`** на недоступную для роли вкладку → откат на «Обзор».

## 14. Критерии готовности

- Один пункт сайдбара «Личный кабинет»; «Настройки» убраны из сайдбара; `/settings` редиректит на профиль-вкладку.
- Заказчик и исполнитель видят корректный свой набор вкладок; переключение работает.
- «Мои заявки»/«В работе» разделяются на «В работе»/«Отработанные».
- ⭐ в «Нормативах» сохраняется и отражается во вкладке «Избранные документы» (переживает перезагрузку).
- Профиль редактируется и сохраняется из кабинета (как раньше в Настройках).
- **Изоляция данных:** кабинет показывает только данные текущего пользователя; ни одна вкладка не раскрывает чужие приватные заявки/отклики/проекты.
- `next build` зелёный; визуальная проверка ключевых вкладок для обеих ролей.

## 15. Открытые вопросы / решаются в плане

- Судьба `FavStrip`/`MOCK_FAVORITES` (синхронизировать со стором или оставить декоративной).
- `cancelled`-заявки: показывать в «Завершённых» или скрывать.
- Хранить активную вкладку в `?tab=` или только в локальном состоянии.

## 16. Права доступа и изоляция данных (cross-cutting, задел на будущее)

**Принцип:** кабинет — приватная зона «только моё». Любая вкладка показывает исключительно данные текущего пользователя:
- **Заказчик** видит только свои размещённые заявки и отклики на них (`getMyOrders` + отклики по своим заявкам). Чужие заявки/отклики в кабинете не показываются.
- **Исполнитель** видит только свои отклики (`getMyResponses`), проекты, где назначен он (`assignedDesignerId === user.id`), своё избранное и свой профиль.
- Состав вкладок и доступные действия определяются ролью (role-based UI) — у заказчика и исполнителя интерфейсы разные.

**Текущее обеспечение (прототип, localStorage):** фильтрация в стор-геттерах по `user.id` + ветвление вкладок по `user.role`. Маркетплейс-разделы (каталоги `/orders`, `/designers`, `/experts`, `/manufacturers`, `/standards`) намеренно остаются публичными для просмотра; приватная зона — только кабинет.

**Задел на будущее (НЕ реализуем сейчас — YAGNI, держим только архитектурный шов):**
- Централизовать решения о доступе в одном месте: карта «роль → возможности» (видимые вкладки/действия) и хелпер `can(user, action, resource)`, а не разбрасывать проверки по компонентам.
- Все обращения к «моим данным» — через стор-геттеры (единая точка), чтобы будущий серверный энфорсмент прав подменялся прозрачно, без переписывания UI.
- Полноценный RBAC и роут-гварды — отдельной итерацией, когда появится бэкенд; сейчас достаточно изоляции данных + role-based вкладок, спроектированных так, чтобы их легко было ужесточить.
